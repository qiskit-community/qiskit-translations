Συνεισφορά στο Qiskit¶

Το Qiskit είναι ένα έργο ανοιχτού κώδικα που έχει δεσμευτεί να φέρει κβαντικούς υπολογιστές σε ανθρώπους όλων των υποβάθρων. Αυτή η σελίδα περιγράφει πώς μπορείτε να συμμετάσχετε στην κοινότητα Qiskit σε αυτόν τον σκοπό.

Χρησιμοποιούμε τα GitHub pull requests για να δεχτούμε συνεισφορές.

Το θέμα μας δίνει τη δυνατότητα να μιλήσουμε για την ιδέα και τον τρόπο με τον οποίο μπορούμε να συνεργαστούμε για την εφαρμογή της στον κώδικα. Υποβολή Μηνυμάτων¶

Το περιεχόμενο του μηνύματος υποβολών, που περιγράφει την αλλαγή είναι εξίσου σημαντικό με την αλλαγή την ίδια. Το μήνυμα υποβολής παρέχει το υπόβαθρο όχι μόνο για την αναθεώρηση του κώδικα, αλλά και για το ιστορικό των αλλαγών στο αρχείο καταγραφής git. Ένα λεπτομερές μήνυμα υποβολής θα διευκολύνει την αναθεώρηση του κώδικά σας, και επίσης θα παρέχει επεξήγηση για την αλλαγή, όταν κάποιος θα το κοιτάξει στο μέλλον. Όταν γράφετε ένα μήνυμα υποβολής, να θυμάστε αυτές τις σημαντικές λεπτομέρειες:

Μην υποθέτετε ότι ο κριτής κατανοεί ποίο ήταν το αρχικό πρόβλημα.

Κατά την ανάγνωση ενός θέματος, μετά από μια σειρά σχολίων, είναι συχνά σαφές ποια είναι η ρίζα του προβλήματος. Το μήνυμα δέσμευσης πρέπει να έχει μια σαφή δήλωση σχετικά με το αρχικό πρόβλημα. Το σφάλμα είναι απλά ένα ενδιαφέρον ιστορικό υπόβαθρο στο * πώς * ταυτοποιήθηκε το πρόβλημα. Θα πρέπει να υπάρχει η δυνατότητα να αναθεωρηθεί ένα προτεινόμενο πακέτο για επιδιόρθωση από το μήνυμα υποβολής, χωρίς να χρειάζεται να διαβαστεί το ticket του σφάλματος.

Μην θεωρείτε ότι ο κώδικας αυτοτεκμηριώνεται.

Αυτό που είναι αυτονόητο σε ένα άτομο, μπορεί να μην είναι σαφές σε κάποιον άλλο. Να τεκμηριώνετε πάντα ποιό ήταν το αρχικό πρόβλημα και πώς καθορίζεται, για οποιαδήποτε αλλαγή, εκτός από τα πιο προφανή τυπογραφικά λάθη ή κενά διαστήματα.

Περιγράψτε γιατί γίνεται μια αλλαγή.

Ένα κοινό λάθος είναι να καταγράψετε μόνο τον τρόπο που γράφτηκε ο κώδικας, χωρίς να περιγράψετε γιατί ο developer επέλεξε να το κάνει με το συγκεκριμένο τρόπο. Σίγουρα, θα πρέπει να περιγράψετε τη συνολική δομή του κώδικα, ιδιαιτέρως για μεγάλες αλλαγής, αλλά κυρίως, να είστε σίγουροι να περιγράψετε την πρόθεση/το κίνητρο πίσω από τις αλλαγές.

Διαβάστε το μήνυμα υποβολής για να δείτε αν περιέχει την βελτιωμένη δομή του κώδικα.

Συχνά όταν περιγράφουμε ένα μεγάλο μήνυμα υποβολής, γίνεται φανερό ότι μια υποβολή θα έπρεπε να χωριστεί σε δύο ή περισσότερα μέρη. Μην φοβάστε να πάτε πίσω και να επανατοποθετήσετε την αλλαγή για να την χωρίσετε σε ξεχωριστά αιτήματα λήψης.

Εξασφαλίστε επαρκείς πληροφορίες για να αποφασίσετε αν θα γίνει αναθεώρηση.

Όταν το GitHub στέλνει ειδοποιήσεις μέσω email για νέες υποβολές pull requests, υπάρχουν ελάχιστες πληροφορίες που συμπεριλαμβάνονται - συνήθως μόνο το μήνυμα υποβολής και η λίστα των αλλαγών των αρχείων. Εξαιτίας του μεγάλου όγκου πακέτων, ένα μήνυμα υποβολής πρέπει να περιέχει επαρκείς πληροφορίες για έναν πιθανό κριτικό, ώστε να μπορεί να βρει το πακέτο που πρέπει να αναθεωρήσει.

Η πρώτη γραμμή της υποβολής είναι η σημαντικότερη.

Στις Git υποβολές, η πρώτη γραμμή του μηνύματος δέσμευσης έχει ιδιαίτερη σημασία. Χρησιμοποιείται ως ο προεπιλεγός τίτλος της αίτησης έλξης, η γραμμή θέματος ειδοποίησης μέσω email, τα μηνύματα σχολιασμού git, οι σημειώσεις της λειτουργίας προβολής gitk, η συγχώνευση μηνυμάτων δέσμευσης και πολλά άλλα σημεία όπου ο χώρος βρίσκεται σε ένα ασφάλιστρο. Καθώς και συνοψίζοντας την ίδια την αλλαγή, θα πρέπει να φροντίσει να λεπτομέρεια ποιο μέρος του κώδικα επηρεάζεται.

In addition, the first line of the commit message becomes an entry in the generated changelog if the PR is tagged as being included in the changelog. It is critically important that you write clear and succinct summary lines.

Describe any limitations of the current code.

If the code being changed still has future scope for improvements, or any known limitations, mention these in the commit message. This demonstrates to the reviewer that the broader picture has been considered, and what tradeoffs have been done in terms of short-term goals versus long-term wishes.

Πολιτική κατάργησης¶

Qiskit users need to know if a feature or an API they rely upon will continue to be supported by the software tomorrow. Knowing under which conditions the project can remove (or change in a backwards-incompatible manner) a feature or API is important to the user. To manage expectations, the following policy is how API and feature deprecation/removal is handled by Qiskit:

1. Features, APIs, or configuration options are marked deprecated in the code. Appropriate DeprecationWarning class warnings will be sent to the user. The deprecated code will be frozen and receive only minimal maintenance (just so that it continues to work as-is).

2. A migration path will be documented for current users of the feature. This will be outlined in the both the release notes adding the deprecation, and the release notes removing the feature at the completion of the deprecation cycle. If feasible, the warning message will also include the migration path. A migration path might be «stop using that feature», but in such cases it is necessary to first judge how widely used and/or important the feature is to users, in order to determine a reasonable obsolescence date.

3. An obsolescence date for the feature will be set. The feature must remain intact and working (although with the proper warning being emitted) in all releases pushed until after that obsolescence date. At the very minimum, the feature (or API, or configuration option) should be marked as deprecated (and continue to be supported) for at least three months of linear time from the release date of the first release to include the deprecation warning. For example, if a feature were deprecated in the 0.9.0 release of Terra, which was released on August 22, 2019, then that feature should still appear in all releases until at least November 22, 2019. Since releases do not occur at fixed time intervals, a deprecation warning may only occur in one release prior to removal.

Note that this delay is a minimum. For significant features, it is recommended that the deprecated feature appears for at least double that time. Also, per the stable branch policy, deprecation removals can only occur during minor version releases; they are not appropriate for backporting.

Deprecation Warnings¶

The proper way to raise a deprecation warning is to use the warn function from the warnings module in the Python standard library. The warning category class should be a DeprecationWarning. An example would be:

import warnings

def foo(input):
    warnings.warn('The qiskit.foo() function is deprecated as of 0.9.0, and '
                  'will be removed no earlier than 3 months after that '
                  'release date. You should use the qiskit.bar() function '
                  'instead.', DeprecationWarning, stacklevel=2)

One thing to note here is the stack_level kwarg on the warn() call. This argument is used to specify which level in the call stack will be used as the line initiating the warning. Typically, stack_level should be set to 2, as this will show the line calling the context where the warning was raised. In the above example, it would be the caller of foo(). If you did not set this, the warning would show that it was caused by the line in the foo() function, which is not helpful for users trying to determine the origin of a deprecated call. However, this value may be adjusted, depending on the call stack and where warn() gets called from. For example, if the warning is always raised by a private method that only has one caller, stack_level=3 might be appropriate.

Stable Branch Policy¶

The stable branch is intended to be a safe source of fixes for high-impact bugs and security issues that have been fixed on master since a release. When reviewing a stable branch PR, we must balance the risk of any given patch with the value that it will provide to users of the stable branch. Only a limited class of changes are appropriate for inclusion on the stable branch. A large, risky patch for a major issue might make sense, as might a trivial fix for a fairly obscure error-handling case. A number of factors must be weighed when considering a change:

  • The risk of regression: even the tiniest changes carry some risk of breaking something, and we really want to avoid regressions on the stable branch.

  • The user visibility benefit: are we fixing something that users might actually notice, and if so, how important is it?

  • How self-contained the fix is: if it fixes a significant issue but also refactors a lot of code, it’s probably worth thinking about what a less risky fix might look like.

  • Whether the fix is already on master: a change must be a backport of a change already merged onto master, unless the change simply does not make sense on master.

Backporting procedure:¶

When backporting a patch from master to stable, we want to keep a reference to the change on master. When you create the branch for the stable PR, use:

$ git cherry-pick -x $master_commit_id

However, this only works for small self-contained patches from master. If you need to backport a subset of a larger commit (from a squashed PR, for example) from master, do this manually. In these cases, add:

Backported from: #master pr number

so that we can track the source of the change subset, even if a strict cherry-pick doesn't make sense.

If the patch you’re proposing will not cherry-pick cleanly, you can help by resolving the conflicts yourself and proposing the resulting patch. Please keep Conflicts lines in the commit message to help review of the stable patch.

Δομή τεκμηρίωσης¶

Ο τρόπος με τον οποίο η τεκμηρίωση είναι δομημένη στο Qiskit είναι να ωθήσει όσο το δυνατόν περισσότερα από τα πραγματικά έγγραφα στα docstrings. Αυτό διευκολύνει τις προσθήκες και τις διορθώσεις κατά τη διάρκεια της ανάπτυξης, διότι η πλειονότητα των εγγράφων ζει κοντά στον κώδικα που αλλάζει. Υπάρχουν τρία επίπεδα στην κανονική δομή τεκμηρίωσης στο Terra:

Τα αρχεία .rst` στα docs/apidocs

These files are used to tell Sphinx which modules to include in the rendered documentation. This contains two pieces of information: an internal reference or cross reference to the module, which can be used for internal links inside the documentation, and an automodule directive used to parse the module docstrings from a specified import path. For example, the dagcircuit.rst file contains:

.. _qiskit-dagcircuit:


.. automodule:: qiskit.dagcircuit
   :no-members:
   :no-inherited-members:
   :no-special-members:

The only .rst file outside of this is qiskit.rst, which contains the table of contents. If you’re adding a new .rst file for a new module’s documentation, make sure to add it to the toctree in that file.

Το docstring επιπέδου λειτουργικής μονάδας

This docstring is at the module level for the module specified in the automodule directive in the rst file. If the module specified is a directory/namespace, the docstring should be specified in the __init__.py file for that directory. This module-level docstring contains more details about the module being documented. The normal structure to this docstring is to outline all the classes and functions of the public API that are contained in that module. This is typically done using the autosummary directive (or autodoc directives directly if the module is simple, such as in the case of qiskit.execute). The autosummary directive is used to autodoc a list of different Python elements (classes, functions, etc.) directly without having to manually call out the autodoc directives for each one. The module-level docstring is where to provide a high-level overview of what functionality the module provides. This is normally done by grouping the different components of the public API together into multiple subsections.

Για παράδειγμα, όπως στην προηγούμενη ενότητα dagcircuit module, τα περιεχόμενα της ενότητας docstring για το `qiskit/dagcircuit/__init__.py θα είναι:

"""
=======================================
DAG Circuits (:mod:`qiskit.dagcircuit`)
=======================================
.. currentmodule:: qiskit.dagcircuit
DAG Circuits
============
.. autosummary::
   :toctree: ../stubs/
   DAGCircuit
   DAGNode
Exceptions
==========
.. autosummary::
   :toctree: ../stubs/
   DAGCircuitError
"""

Σημείωση

This is just an example and the actual module docstring for the dagcircuit module might diverge from this.

The actual docstring for the elements listed in the module docstring

Θα πρέπει να προσπαθήσετε να τεκμηριώσετε λεπτομερώς όλες τις δημόσιες διεπαφές που εκτίθενται χρησιμοποιώντας παραδείγματα, όταν είναι απαραίτητο. Για docstrings, χρησιμοποιούνται Google Python Style Docstrings. Αυτό αναλύεται με την επέκταση napoleon sphinx. Το napoleon documentation περιέχει ένα καλό παράδειγμα για το πώς θα πρέπει να διαμορφώνονται οι docstrings.

Σημείωση

You can use any Sphinx directive or rst formatting in a docstring as it makes sense. For example, one common extension used is the jupyter-execute directive, which is used to execute a code block in Jupyter and display both the code and output. This is particularly useful for visualizations.

Ενσωμάτωση Τεκμηρίωσης¶

The hosted documentation at https://qiskit.org/documentation/ covers the entire Qiskit project; Terra is just one component of that. As such, the documentation builds for the hosted version are built by the Qiskit meta-package repository https://github.com/Qiskit/qiskit. When commits are merged to that repo, the output of Sphinx builds are uploaded to the qiskit.org website. Those Sphinx builds are configured to pull in the documentation from the version of the Qiskit elements installed by the meta-package at that point. For example, if the meta-package version is currently 0.13.0, then that will copy the documentation from Terra’s 0.10.0 release. When the meta-package’s requirements are bumped, then it will start pulling documentation from the new version. This means that fixes for incorrect API documentation will need to be included in a new release. Documentation fixes are valid backports for a stable patch release per the stable branch policy (see that section below).

Κατά τη διάρκεια της διαδικασίας κατασκευής, τα περιεχόμενα του docs/apidocs/ του κάθε στοιχείου αντιγράφονται αναδρομικά σε ένα κοινόχρηστο αντίγραφο του doc/apidocs/ στο αποθετήριο μετα-πακέτου μαζί με όλα τα άλλα στοιχεία. Αυτό σημαίνει ότι αυτό που βρίσκεται στη ρίζα των docs/apidocs σε κάθε στοιχείο σε μια έκδοση θα καταλήξει στη ρίζα του https://qiskit.org/documentation/apidoc/.

Η εγκατάσταση από την πηγή απαιτεί να έχετε ένα μεταγλωττιστή C++ στο σύστημά σας που υποστηρίζει C++11.

Στις περισσότερες πλατφόρμες Linux, ο απαραίτητος μεταγλωττιστής GCC έχει ήδη εγκατασταθεί.

Αν χρησιμοποιείτε macOS, μπορείτε να εγκαταστήσετε τον μεταγλωττιστή Clang εγκαθιστώντας το XCode. Ελέγξτε αν έχετε εγκαταστήσει το XCode και το Clang, ανοίγοντας το τερματικό παράθυρο και εισάγοντας τα ακόλουθα.

clang --version

Εγκαταστήστε το XCode και Clang χρησιμοποιώντας την ακόλουθη εντολή.

xcode-select --install

Στα Windows, είναι ευκολότερο να εγκαταστήσετε τον μεταγλωττιστή Visual C++ από τα Εργαλεία Κατασκευής για το Visual Studio 2017. Αντ “αυτού μπορείτε να εγκαταστήσετε το Visual Studio έκδοση 2015 ή 2017, φροντίζοντας να επιλέξετε τις επιλογές για την εγκατάσταση του μεταγλωττιστή C++.

Μόλις εγκατασταθούν οι μεταγλωττιστές, είστε έτοιμοι να εγκαταστήσετε το Qiskit Terra.

  1. Κλωνοποιήστε το αποθετήριο Terra.

    git clone https://github.com/Qiskit/qiskit-terra.git

  2. Η κλωνοποίηση του αποθετηρίου δημιουργεί έναν τοπικό φάκελο που ονομάζεται qiskit-terra.

    cd qiskit-terra

  3. Εγκαταστήστε τις βιβλιοθήκες απαιτήσεων Python από τον κατάλογο qiskit-terra.

    pip install cython

  4. Αν θέλετε να εκτελέσετε δοκιμές ή να επιβάλετε ελέγχους, εγκαταστήστε τις απαιτήσεις ανάπτυξης.

    pip install -r requirements-dev.txt

  5. Εγκατάσταση ` ` qiskit-terra ` `.

    pip install .

Αν θέλετε να το εγκαταστήσετε σε επεξεργάσιμη μορφή, που σημαίνει ότι οι αλλαγές κώδικα στο έργο δεν απαιτούν την εφαρμογή επανεγκατάστασης, μπορείτε να το κάνετε με:

pip install -e .

Στη συνέχεια μπορείτε να εκτελέσετε τα παραδείγματα κώδικα μετά την εγκατάσταση του Terra. Μπορείτε να εκτελέσετε το παράδειγμα με την ακόλουθη εντολή.

python examples/python/using_qiskit_terra_level_0.py

Σημείωση

Αν δεν σκοπεύετε να εγκαταστήσετε άλλα στοιχεία, το qiskit-terra θα εκπέμπει μια προειδοποίηση RuntimeWarning καθώς το qiskit-aer και qiskit-ibmq-δεν είναι εγκατεστημένα. Αυτό γίνεται επειδή οι χρήστες συνήθως προτίθενται να χρησιμοποιήσουν τα πρόσθετα στοιχεία, αλλά δεν συνειδητοποιούν ότι δεν έχουν εγκατασταθεί, ή ότι η εγκατάσταση είτε της Aer είτε του IBM Quantum Provider απέτυχε για κάποιο λόγο. Αν θέλετε να καταργήσετε αυτές τις προειδοποιήσεις, προσθέστε:

    dnf install @development-tools

    apt-get install build-essential

    dnf install openblas-devel

    apt-get install libopenblas-dev

    Για να χρησιμοποιήσετε τον μεταγλωττιστή “Clang <https://clang.llvm.org/>” σε macOS, θα πρέπει να εγκαταστήσετε μία επιπρόσθετη βιβλιοθήκη για την υποστήριξη του “OpenMP <https://www.openmp.org/>”.     brew install libomp

  2. Στη συνέχεια εγκαταστήστε μια υλοποίηση BLAS. Το “ OpenBLAS <https://www.openblas.net/>` __ είναι η προεπιλεγμένη επιλογή.

    brew install openblas

    xcode-select --install

  1. Εγκατάσταση απαιτήσεων μεταγλωττιστή.

    conda install --update-deps vs2017_win-64 vs2017_win-32 msvc_runtime

  1. Κατασκευάστε και εγκαταστήστε απευθείας το qiskit-aer

    Εάν έχετε εγκαταστήσει το pip <19.0.0 και το περιβάλλον σας δεν απαιτεί προσαρμοσμένη κατασκευή, εκτελέστε:

    cd qiskit-aer

Προσαρμοσμένες επιλογές κατά την κατασκευή του wheel¶

Το σύστημα κατασκευής Aer χρησιμοποιεί scikit-build για να εκτελέσει τη μεταγλώττιση κατά τη δημιουργία του με τη διεπαφή Python. Δρα ως διεπαφή για τα setuptools για να καλέσετε CMake και να μεταγλωττίσετε τις εφαρμογές για το τοπικό σας σύστημα.

Λόγω της πολυπλοκότητας της κατάρτισης των εφαρμογών, μπορεί να χρειαστεί να περάσετε τις επιλογές σε ένα ορισμένο μέρος της διαδικασίας κατασκευής. Ο τρόπος για να περάσετε μεταβλητές είναι:

python setup.py bdist_wheel [skbuild_opts] [-- [cmake_opts] [-- build_tool_opts]]

όπου τα στοιχεία μέσα στις αγκύλες [] είναι προαιρετικά και skbuild_opts, cmake_opts`, build_tool_opts πρέπει να αντικατασταθούν από σημαίες της επιλογής σας. Μια λίστα επιλογών CMake είναι διαθέσιμη εδώ: https://cmake.org/cmake/help/v3.6/manual/cmake.1.html#options. Για παράδειγμα, θα μπορούσατε να εκτελέσετε κάτι όπως:

python setup.py bdist_wheel -- -- -j8

Αυτό περνάει τη σημαία -j8 στο υποκείμενο σύστημα κατασκευής (το οποίο σε αυτή την περίπτωση είναι Automake), λέγοντάς του ότι θέλετε να οικοδομήσουμε παράλληλα χρησιμοποιώντας 8 διαδιακασίες.

Για παράδειγμα, μια κοινή περίπτωση χρήσης για αυτές τις σημαίες στο linux είναι να καθορίσετε μια συγκεκριμένη έκδοση του μεταγλωττιστή C++ που θα χρησιμοποιηθεί (κανονικά αν η προεπιλογή είναι πολύ παλιά):

python setup.py bdist_wheel -- -DCMAKE_CXX_COMPILER=g++-7

που θα πει στο CMake να χρησιμοποιεί την εντολή g++-7 αντί της προκαθορισμένης g++ κατά την μεταγλώττιση του Aer.

Μια άλλη συνηθισμένη περίπτωση χρήσης για αυτό, ανάλογα με το περιβάλλον σας, είναι ότι μπορεί να χρειαστεί να καθορίσετε το όνομα της πλατφόρμας σας και να απενεργοποιήσετε τη στατική σύνδεση.

python setup.py bdist_wheel --plat-name macosx-10.9-x86_64 \
-- -DSTATIC_LINKING=False -- -j8

Εδώ το --plat-name είναι μια σημαία στο setuptools, για να καθορίσετε το όνομα της πλατφόρμας που θα χρησιμοποιηθεί στα μεταδεδομένα του πακέτου, -DSTATIC_LINKING είναι μια σημαία για τη χρήση CMake για την απενεργοποίηση στατικής σύνδεσης, και -j8 είναι μια σημαία για τη χρήση του Automake για τη χρήση 8 διαδικασιών για μεταγλώττιση.

Εγκατάσταση του Ignis από την πηγή¶

  1. Κλωνοποιήστε το αποθετήριο του Ignis.

    git clone https://github.com/Qiskit/qiskit-ignis.git

  2. Η κλωνοποίηση του αποθετήριου δημιουργεί ένα τοπικό φάκελο με το όνομα qiskit-ignis.

    cd qiskit-ignis

  3. Αν θέλετε να εκτελέσετε δοκιμές ή linting ελέγχους, εγκαταστήστε τις απαιτήσεις του προγραμματιστή. Αυτό δεν απαιτείται για την εγκατάσταση ή τη χρήση του πακέτου qiskit-ignis κατά την εγκατάσταση από την πηγή.

    pip install -r requirements-dev.txt

  4. Εγκαταστήστε το Ignis.

    pip install .

Αν θέλετε να το εγκαταστήσετε σε τροποποιήσιμη μορφή, η οποία σημαίνει ότι οι αλλαγές του κώδικα στο έργο δεν απαιτούν να εφαρμοστεί κάποια επανεγκατάσταση:

pip install -e .
