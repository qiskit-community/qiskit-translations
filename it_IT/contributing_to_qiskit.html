Contribuire a Qiskit¶

Qiskit è un progetto open-source il cui obiettivo è quello di rendere accessibile la computazione quantistica a persone con background differenti. Questa pagina descrive come puoi aiutare la comunità di Qiskit a raggiungere questo obiettivo.

Contribuire al Codice¶

Guida allo Stile¶

Per assicurare uno stile coerente nel codice del progetto, utilizziamo Pylint e pycodesytle per verificare che i contributi siano conformi e rispettino le linee guida di stile del codice. Per verificare che le tue modifiche aderiscono alle linee guida di stile, esegui: tox -elint

Contributor License Agreement¶

Prima di poter sottomettere qualsiasi codice, ogni contributore deve firmare un contributor license agreement (CLA). Con la sottoscrizione del CLA, si dichiara di essere l’autore del codice e che si sta contribuendo liberamente sotto i termini della licenza Apache-2.0.

Quando si contribuisce al Qiskit con una nuova richiesta di pull, un bot verificherà che sia stato firmato o meno il CLA. Se necessario, il bot lascerà un commento sulla richiesta di pull, contenente un link per accettare le condizioni. Il documento individual CLA è disponibile per visione in formato PDF.

Nota

Se il tuo contributo è parte del tuo lavoro o di proprietà del tuo datore di lavoro, allora dovresti firmare anche la corporate CLA ed inviarla al seguente indirizzo <qiskit@us.ibm.com>.

Richieste di Pull¶

Utilizziamo GitHub pull requests per accettare contributi.

Anche se non è richiesto, aprire una nuova issue sul bug che stai correggendo o sulla funzionalità su cui stai lavorando prima di aprire una richiesta di pull è un passo importante per iniziare una discussione con la comunità sul tuo lavoro. Aprire una issue ci dà la possibilità di poter parlare dell’idea e di come si può lavorare insieme per implementarla nel codice. Inoltre, consente alla comunità di sapere su cosa stai lavorando e, se hai bisogno di aiuto, puoi fare riferimento all’issue quando ne discuti con altri membri della comunità e del team.

Se hai scritto del codice ma hai bisogno di aiuto per finirlo, vuoi avere un feedback iniziale su di esso prima di finirlo, o vuoi condividerlo e discuterne prima di completare l’implementazione, puoi aprire una richiesta di pull Work in Progress. Quando crei la richiesta di pull, inserisci in testa al titolo il tag [WIP] (per Work In Progress). Questo indicherà ai revisori che il codice nella richiesta di pull non è nel suo stato finale e verrà modificato. Significa anche che non faremo il merge del commit fino a quando non sarà terminato. Tu o un revisore potete rimuovere il tag [WIP] quando il codice è pronto per essere completamente revisionato per il suo inserimento.

Messaggi di Commit¶

Il contenuto del messaggio di commit che descrive una modifica è importante tanto quanto la modifica stessa. Il messaggio di commit fornisce informazioni utili non solo alla revisione del codice ma anche al tracciamento delle modifiche nel git log. Un messaggio di commit dettagliato renderà più facile la revisione del codice e fornirà anche il contesto della modifica quando qualcuno la guarderà in futuro. Quando si scrive un messaggio di commit, è importante ricordare questi importanti dettagli:

Non dare per scontato che il recensore capisca quale fosse il problema originale.

Quando si legge una issue, dopo una serie di commenti, è molto spesso chiaro quale sia il problema alla radice. Il messaggio di commit dovrebbe contenere una chiara dichiarazione su quale fosse il problema originale. Il bug fornisce solamente il contesto storico su come il problema è stato identificato. Dovrebbe essere possibile poter revisionare una patch proposta per verificarne la correttezza a partire dalla lettura del messaggio di commit, senza bisogno di leggere il ticket del bug.

Non dare per scontato che il codice sia ovvio o auto-esplicativo.

Ciò che è ovvio per una persona, potrebbe non essere chiaro per un’altra. É opportuno documentare qual è stato il problema originale e come è stato risolto, includendo qualsiasi modifica effettuata, a parte gli errori di battitura più evidenti o le commit che contengono solo spazi bianchi.

Descrivere perché si sta apportando la modifica.

Un errore comune è quello di documentare solo come il codice è stato scritto, senza descrivere perché lo sviluppatore ha scelto di farlo in questo modo. É sicuramente importante descrivere la struttura generale del codice, in particolare se si tratta di grandi modifiche, ma è ancora più importante descrivere l’intento/motivazione dietro le modifiche.

Leggi il messaggio di commit per vedere se accenna a una struttura del codice migliorata.

Spesso quando si descrive un messaggio di commit molto corposo, ci si potrebbe rendere conto che sarebbe stato più opportuno suddividere la commit in due o più parti. Non aver paura di tornare indietro e riorganizzare la modifica per suddividerla in richieste di pull separate.

Assicurare informazioni sufficienti per decidere se procedere alla revisione.

Quando GitHub invia gli avvisi per email sull’apertura di nuove richieste di pull, le informazioni incluse sono minime - di solito solo il messaggio di commit e la lista delle modifiche nei file. A causa dell’alto volume di patch, un messaggio di commit deve contenere informazioni sufficienti per permettere ai potenziali revisori di trovare la patch da revisionare.

La prima riga di commit è la più importante.

Nelle commit di Git, la prima riga del messaggio ha un significato speciale. Viene infatti utilizzata come titolo predefinito della richiesta di pull, riga dell’oggetto dell’email di notifica, messaggi di annotazione di git, annotazioni di gitk viewer, messaggi di merge di commit, ed in molti altri posti dove lo spazio è prezioso. Oltre a riassumere la modifica stessa, si dovrebbe fare attenzione a dettagliare quale parte del codice è interessata.

Inoltre, la prima riga del messaggio di commit diventa una voce nel changelog generato se la PR è etichettata come inclusa nel changelog. È di fondamentale importanza scrivere righe di riepilogo chiare e succinte.

Descrivere le possibili limitazioni del codice attuale.

Se il codice che viene modificato ha ancora margini di miglioramento futuri, o eventuali limitazioni note, menzionarle nel messaggio di commit. Questo dimostra al revisore che il quadro più ampio è stato preso in considerazione, e quali compromessi sono stati fatti in termini di obiettivi a breve termine rispetto ai piani a lungo termine.

Includere riferimenti alle issue.

Se le correzioni del commit sono relative ad una issue, assicurarsi di annotarla nel messaggio di commit. Usare la sintassi:

Fixes #1234

se risolve la issue (GitHub chiuderà la issue dopo il merge della PR).

La regola principale da seguire è:

Il messaggio di commit deve contenere tutte le informazioni necessarie per comprendere appieno la patch e verificarne la correttezza. Essere eccessivamente sintetici non è la scelta migliore.

Documenta il Tuo Codice¶

Se si effettua una modifica ad un elemento, assicurarsi di aggiornare le docstrings associate e le parti della documentazione sotto docs/apidocs nella repository corrispondente. Per costruire localmente la documentazione specifica dell’elemento, si esegua tox -edocs per compilare e costruire la documentazione localmente e salvare l’output in docs/_build/html. In aggiunta, il comando eseguito dal job Docs CI sulla pipeline azure creerà un file zip dell’output che si potrà scaricare e visualizzare localmente.

Se si ha una issue con la documentazione combinata che è manutenuta nel Qiskit/qiskit repo, è possibile aprire una issue di documentazione nel caso in cui si individuano bug di documentazione, se una nuova funzionalità deve essere documentata, o se si pensa che si possa aggiungere del materiale ai documenti esistenti.

Primi Validi Contributi¶

Se si vuole contribuire a Qiskit, ma non si è sicuri di dove iniziare, l’etichetta «good first issue» sulle issue di un progetto mette in evidenza gli elementi appropriati per le persone nuove al progetto. Queste sono tutte issue che sono state esaminate ed etichettate dai collaboratori come qualcosa su cui un nuovo collaboratore dovrebbe essere in grado di lavorare. In altre parole, l’intima familiarità con Qiskit non è un requisito per sviluppare una soluzione per tali issue.

Politica di deprecazione¶

Gli utenti di Qiskit devono sapere se una funzionalità o un’API che utilizzano continuerà ad essere supportata dal software anche in futuro. Sapere in quali condizioni il progetto può rimuovere (o modificare in modo non retro compatibile) una funzionalità o un’API è importante per l’utente. Per gestire le aspettative, la seguente politica descrive come vengono gestite da Qiskit la deprecazione/rimozione di API e funzionalità:

1. Features, APIs, or configuration options are marked deprecated in the code. Appropriate DeprecationWarning class warnings will be sent to the user. The deprecated code will be frozen and receive only minimal maintenance (just so that it continues to work as-is).

2. A migration path will be documented for current users of the feature. This will be outlined in the both the release notes adding the deprecation, and the release notes removing the feature at the completion of the deprecation cycle. If feasible, the warning message will also include the migration path. A migration path might be «stop using that feature», but in such cases it is necessary to first judge how widely used and/or important the feature is to users, in order to determine a reasonable obsolescence date.

3. An obsolescence date for the feature will be set. The feature must remain intact and working (although with the proper warning being emitted) in all releases pushed until after that obsolescence date. At the very minimum, the feature (or API, or configuration option) should be marked as deprecated (and continue to be supported) for at least three months of linear time from the release date of the first release to include the deprecation warning. For example, if a feature were deprecated in the 0.9.0 release of Terra, which was released on August 22, 2019, then that feature should still appear in all releases until at least November 22, 2019. Since releases do not occur at fixed time intervals, a deprecation warning may only occur in one release prior to removal.

Si noti che questo ritardo è minimo. Per le funzionalità significative, si raccomanda che la funzione deprecata appaia almeno per il doppio del tempo. Inoltre, secondo la politica di «stable branch», la rimozione della deprecazione può avvenire solo durante i rilasci di versioni minori; non è appropriata per il backporting.

Avvisi di deprecazione¶

Il modo corretto per segnalare un avviso di deprecazione è quello di utilizzare la funzione warn dal modulo warnings nella libreria standard Python. La classe della categoria di avviso dovrebbe essere un DeprecationWarning. Un esempio sarebbe:

import warnings

def foo(input):
    warnings.warn('The qiskit.foo() function is deprecated as of 0.9.0, and '
                  'will be removed no earlier than 3 months after that '
                  'release date. You should use the qiskit.bar() function '
                  'instead.', DeprecationWarning, stacklevel=2)

Una cosa da notare qui è il kwarg stack_level sulla chiamata warn(). Questo argomento è usato per specificare quale livello nello stack di chiamate sarà indicato come linea di avvio dell’avviso. Tipicamente, stack_level dovrebbe essere impostato a 2, in quanto questo mostrerà la linea che chiama il contesto dove è stato generato l’avviso. Nell’esempio precedente, sarebbe il chiamante di foo(). Se non lo si imposta, l’avviso mostrerebbe che è stato causato dalla linea nella funzione foo(), il che non è utile per gli utenti che cercano di determinare l’origine di una chiamata deprecata. Tuttavia, questo valore può essere regolato, a seconda dello stack di chiamate e da dove warn() viene chiamato. Per esempio, se l’avviso è sempre generato da un metodo privato che ha un solo chiamante, potrebbe essere appropriato stack_level=3.

Linea di condotta sullo Stable Branch¶

Lo stable branch è pensato per essere una fonte sicura delle correzioni per bug ad alto impatto e per problemi di sicurezza che sono stati risolti sul master a valle di un rilascio. Quando controlliamo una richiesta di pull sullo stable branch, dobbiamo bilanciare il rischio associato a ciascuna patch con il valore che apporta agli utilizzatori dello stable branch stesso. Solo poche modifiche sono adatte ad essere incluse nello stable branch. Una patch estesa e rischiosa per un problema importante può avere senso, così come può averlo una banale modifica per un caso raro di gestione degli errori. Vari fattori vanno presi in considerazione quando si valuta una modifica:

  • Il rischio di regressione: anche nella più piccola modifica c’è il rischio di rompere qualcosa, e noi vogliamo davvero evitare di fare regressioni sullo stable branch.

  • Il vantaggio osservabile dall’utente: stiamo correggendo qualcosa che gli utenti noteranno, e, se sì, quanto è importante?

  • Quanto è auto-contenuta la correzione: se corregge una issue importante, ma modifica anche molto codice, probabilmente conviene pensare a come potrebbe essere implementata una correzione meno rischiosa.

  • Se la correzione è già nel master: una modifica deve essere un backport di una modifica già unita al master, a meno che la modifica semplicemente non abbia senso sul master.

Procedura di backporting:¶

Quando si fa backporting di una patch dal master allo stable branch, è bene mantenere una referenza alla modifica sul master. Quando si crea il branch per la richiesta di pull sullo stable, si deve usare:

$ git cherry-pick -x $master_commit_id

Tuttavia, questo funziona solo per patch dal master piccole ed auto-contenute. Se serve fare backport dal master di una parte di una commit più grande (da una richiesta di pull squashed, ad esempio), va fatto manualmente. In questi casi va aggiunto:

Backported from: #master pr number

cosicchè è possibile tracciare la sorgente di quella parte di modifica, anche se un rigoroso cherry-pick non dà risultati utili.

Se sulla patch che si propone il cherry-pick non risulta pulito, si può dare supporto risolvendo i conflitti e proponendo la patch risultante. E” utile mantenere le linee relative al conflitto nel messaggio di commit in modo da aiutare durante la revisione della patch dello stable.

Etichette di backport¶

Bug o richieste di pull etichettate come stable backport potential sono bug che interessano anche lo stable e che potrebbero essere idonei al backport una volta che una correzione viene unita nel master. Una volta proposto il backport, l’etichetta deve essere rimossa.

Si deve includere [Stable] nel titolo della richiesta di pull verso lo stable branch, come indicazione che la scelta dello stable come branch di arrivo non è un errore. Inoltre, va referenziato il numero della richiesta di pull del master che si sta portando.

Contribuire alla Documentazione¶

La documentazione di Qiskit è conforme alla filosofia di docs as code, principalmente rielaborata a partire dai commenti al codice di Qiskit seguendo style of API documentation.

La documentazione è costruita a partire dal master branch di Qiskit/qiskit/docs usando Sphinx. La maggior parte della documentazione, che si trova in API Reference, è costruita a partire dai commenti al codice nelle repository elencate in Dove Sono Le Cose.

Struttura della Documentazione¶

La documentazione in Qiskit è strutturata in modo da effettuare il più possibile la push della reale documentazione nelle docstring. Questo rende più facile fare aggiunte e correzioni durante lo sviluppo, perché la maggior parte della documentazione è vicina al codice che si sta modificando. Ci sono tre livelli nella struttura normale della documentazione in Terra:

I file .rst in docs/apidocs

Questi file vengono utilizzati per indicare a Sphinx quali moduli utilizzare nella documentazione generata. Questo contiene due tipi di informazioni: un riferimento interno o riferimento incrociato al modulo, il quale può essere utilizzato per i link interni alla documentazione stessa, ed una direttiva automodule utilizzata per analizzare le docstrings del modulo a partire da un percorso di importazione (import path). Ad esempio, il file dagcircuit.rst contiene:

.. _qiskit-dagcircuit:


.. automodule:: qiskit.dagcircuit
   :no-members:
   :no-inherited-members:
   :no-special-members:

Il file qiskit.rst è a parte e contiene la tabella dei contenuti. Se stai aggiungendo un nuovo file .rst per la documentazione di un nuovo modulo, assicurati di aggiungerlo al file toctree.

La docstring a livello modulo

Questa docstring è a livello modulo per il modulo specificato nell’istruzione automodule all’interno del file rst. Se il modulo specificato è una cartella/namespace, la docstring deve essere specificata nel file __init__.py di quella cartella. Questa doctring a livello modulo contiene più dettagli sul modulo che si sta documentando. La struttura normale per questa docstring serve a delineare tutte le classi e le funzioni dell’API pubblica che sono contenute nel modulo. Questo viene fatto generalmente usando l’istruzione autosummary (o direttamente autodoc directives se il modulo è semplice, come nel caso di qiskit.execute). L’istruzione autosummary è utilizzata per auto-documentare direttamente una lista di elementi di Python (classi, funzioni, etc..) senza dover chiamare manualmente l’autodoc per ciascuno. La docstring a livello modulo è il luogo adatto per inserire una visione ad alto livello delle funzionalità che il modulo fornisce. Questo viene fatto generalmente raggruppando le diverse componenti dell’API pubblica insieme in varie sottosezioni.

Ad esempio, nel modulo dagcircuit precedentemente descritto, il contenuto del modulo docstring per qiskit/dagcircuit/__init__.py sarebbe:

"""
=======================================
DAG Circuits (:mod:`qiskit.dagcircuit`)
=======================================
.. currentmodule:: qiskit.dagcircuit
DAG Circuits
============
.. autosummary::
   :toctree: ../stubs/
   DAGCircuit
   DAGNode
Exceptions
==========
.. autosummary::
   :toctree: ../stubs/
   DAGCircuitError
"""

Nota

Questo è solo un esempio e il modulo docstring reale per il modulo del dagcircuit potrebbe essere differente.

La docstring effettiva per gli elementi elencati nella docstring del modulo

Bisogna sforzarsi di documentare accuratamente tutte le interfacce pubbliche esposte usando esempi laddove necessario. Per le docstring, viene usata la Google Python Style Docstrings. Questo è interpretato usando napoleon sphinx extension. La documentazione napoleon contiene un buon esempio di come dovrebbero essere formattate le docstring.

Nota

È possibile utilizzare qualsiasi direttiva Sphinx o formattazione rst in una docstring se essa ha senso. Ad esempio, un’estensione comunemente utilizzata è la direttiva jupyter-execute, utilizzata per eseguire un blocco di codice in Jupyter e visualizzare sia il codice che l’output. Ciò è particolarmente utile per le visualizzazioni.

Integrazione della Documentazione¶

La documentazione che si trova su https://qiskit.org/documentation/ riguarda l’intero progetto Qiskit; Terra è solo un suo componente. Pertanto, le build della documentazione per la versione ospitata vengono costruite a partire dalla repository dei meta-package di Qiskit https://github.com/Qiskit/qiskit. Quando le commit sono uniti a quella repository, gli output delle Sphinx build vengono caricati sul sito qiskit.org. Queste build Sphinx sono configurate per fare pull della documentazione dalla versione degli elementi di Qiskit installati dal meta-package in quel momento. Per esempio, se la versione del meta-package attuale è la 0.13.0, allora verrà copiata la documentazione dal rilascio 0.10.0 di Terra. Quando i requisiti del meta-package vengono aggiornati, allora viene scaricata anche la documentazione dalla nuova versione. Questo implica che le correzioni per gli errori sulla documentazione dell’API devono essere inclusi in un nuovo rilascio. Le correzioni alla documentazione sono backport adatti ad una patch dello stable branch secondo le linee guida dello stable branch (si veda sotto la sezione dedicata).

Durante il processo di build, il contenuto di ciascun elemento docs/apidocs/ è copiato ricorsivamente in una copia condivisa di doc/apidocs/ nel repository dei meta-package insieme a tutti gli altri elementi. Ciò significa che il contenuto della cartella docs/apidocs finirà nella root di https://qiskit.org/documentation/apidoc/.

Tradurre la Documentazione¶

La documentazione di Qiskit è tradotta (localizzata) usando Crowdin, una piattaforma software e web di localizzazione che consente alle organizzazioni di coordinare progetti di traduzione e collaborare con le comunità per la traduzione dei materiali. Crowdin permette alla nostra comunità di traduttori di amplificare il loro impatto riutilizzando automaticamente il lavoro investito nella traduzione di una frase per tradurre altre frasi simili. Crowdin rende inoltre le traduzioni flessibili a più tipi di modifiche al materiale originale, come lo spostamento di frasi, anche tra file.

Le richieste di localizzazione di Qiskit sono gesite nel repository Qiskit Translations. Per contribuire alla localizzazione di Qiskit, si prega di seguire i seguenti passaggi:

  1. Aggiungere il proprio nome (o ID) al file LOCALIZATION_CONTRIBUTORS.

  2. Creare una Richiesta di Pull per effettuare la merge delle modifiche apportate. Assicurarsi di seguire il modello presente per aprire una Richiesta di Pull.

    Nota

    • Ogni collaboratore deve creare la propria richiesta di pull e firmare il CLA.

    • Si prega di menzionare la lingua a cui si desidera contribuire nel sommario della richiesta di pull.

    • Se ci sono issue aperte per una richiesta di localizzazione, aggiungere il link della issue alla richiesta di pull.

  3. Verrà richiesto di firmare il Qiskit Contributors License Agreement (CLA); si prega di farlo.

  4. Affinché un nuovo linguaggio venga aggiunto ed ottenga il supporto ufficiale da parte degli amministratori del progetto di localizzazione, è necessario un minimo di tre collaboratori.

  5. Among the group of contributors, a translation lead must be identified to serve as a liaison with the administrators of the localization project. The lead must contact: Yuri Kobayashi (yurik@jp.ibm.com) by email.

  6. In the Qiskit-Docs Crowdin project, choose the language that you want to contribute to.

    Nota

    As mentioned in the blog post, Qiskit in my language is Qiskit, we want to make sure that translated languages have enough community support to build a translation team with translators, proofreaders, and translation leads. If you want to be a translation lead or would be willing to join a new translation project team, you can open a GitHub issue to start a discussion with the Qiskit team and recruit translation project members.

  7. Click the Join button and paste the URL of your PR in the dialog box where you are asked why you want to join the Crowdin project.

The administrators of the Crowdin project will review your request and give you access as quickly as they can.

Building from Source¶

You can build a local copy of the documentation from your local clone of the Qiskit/qiskit repository as follows:

  1. Clone the Qiskit repository.

    git clone https://github.com/Qiskit/qiskit.git

  2. Cloning the repository creates a local folder called qiskit.

    cd qiskit

  3. Build the documentation by navigating to your local clone of Qiskit/qiskit and running the following command in a terminal window.

    tox -edocs

    If you do not already have the tox command installed, install it by running:

    pip install tox

As you make changes to your local RST files, you can update your HTML files by navigating to /doc/ and running the following in a terminal window:

tox -edocs

This will build a styled, HTML version of your local documentation repository in the subdirectory /docs/_build/html/.

Installing from Source¶

Installing the elements from source allows you to access the most recently updated version of Qiskit instead of using the version in the Python Package Index (PyPI) repository. This will give you the ability to inspect and extend the latest version of the Qiskit code more efficiently.

When installing the elements and components from source, by default their development version (which corresponds to the master git branch) will be used, as opposed to the stable version (which contains the same codebase as the published pip packages). Since the development versions of an element or component usually include new features and changes, they generally require using the development version of the rest of the items as well.

Nota

The Terra and Aer packages both require a compiler to build from source before you can install. Ignis, Aqua, and the IBM Quantum Provider backend do not require a compiler.

Installing elements from source requires the following order of installation to prevent installing versions of elements that may be lower than those desired if the pip version is behind the source versions:

  1. qiskit-terra

  2. qiskit-aer

  3. qiskit-ignis

  4. qiskit-aqua

  5. qiskit-ibmq-provider (if you want to connect to the IBM Quantum devices or online simulator)

To work with several components and elements simultaneously, use the following steps for each element.

Nota

Due to the use of namespace packaging in Python, care must be taken in how you install packages. If you’re planning to install any element from source, do not use the qiskit meta-package. Also, follow this guide and use a separate virtual environment for development. If you do choose to mix an existing installation with your development, refer to https://github.com/pypa/sample-namespace-packages/blob/master/table.md for the set of combinations of installation methods that work together.

Set up the Virtual Development Environment¶

conda create -y -n QiskitDevenv python=3
conda activate QiskitDevenv

Installing Terra from Source¶

Installing from source requires that you have a C++ compiler on your system that supports C++11.

On most Linux platforms, the necessary GCC compiler is already installed.

If you use macOS, you can install the Clang compiler by installing XCode. Check if you have XCode and Clang installed by opening a terminal window and entering the following.

clang --version

Install XCode and Clang by using the following command.

xcode-select --install

On Windows, it is easiest to install the Visual C++ compiler from the Build Tools for Visual Studio 2017. You can instead install Visual Studio version 2015 or 2017, making sure to select the options for installing the C++ compiler.

Once the compilers are installed, you are ready to install Qiskit Terra.

  1. Clone the Terra repository.

    git clone https://github.com/Qiskit/qiskit-terra.git

  2. Cloning the repository creates a local folder called qiskit-terra.

    cd qiskit-terra

  3. Install the Python requirements libraries from your qiskit-terra directory.

    pip install cython

  4. If you want to run tests or linting checks, install the developer requirements.

    pip install -r requirements-dev.txt

  5. Install qiskit-terra.

    pip install .

If you want to install it in editable mode, meaning that code changes to the project don’t require a reinstall to be applied, you can do this with:

pip install -e .

You can then run the code examples after installing Terra. You can run the example with the following command.

python examples/python/using_qiskit_terra_level_0.py

Nota

If you do not intend to install any other components, qiskit-terra will emit a RuntimeWarning warning that both qiskit-aer and qiskit-ibmq-provider are not installed. This is done because users commonly intend to use the additional elements, but do not realize they are not installed, or that the installation of either Aer or the IBM Quantum Provider failed for some reason. If you wish to suppress these warnings, add:

import warnings
warnings.filterwarnings('ignore', category=RuntimeWarning,
                        module='qiskit')

before any qiskit imports in your code. This will suppress the warning about the missing qiskit-aer and qiskit-ibmq-provider, but will continue to display any other warnings from qiskit or other packages.

Installing Aer from Source¶

  1. Clone the Aer repository.

    git clone https://github.com/Qiskit/qiskit-aer

  2. Install build requirements.

    pip install cmake scikit-build cython

After this, the steps to install Aer depend on which operating system you are using. Since Aer is a compiled C++ program with a Python interface, there are non-Python dependencies for building the Aer binary which can’t be installed universally depending on operating system.

  1. Install compiler requirements.

    Building Aer requires a C++ compiler and development headers.

    If you’re using Fedora or an equivalent Linux distribution, install using:

    dnf install @development-tools

    For Ubuntu/Debian install it using:

    apt-get install build-essential

  2. Install OpenBLAS development headers.

    If you’re using Fedora or an equivalent Linux distribution, install using:

    dnf install openblas-devel

    For Ubuntu/Debian install it using:

    apt-get install libopenblas-dev

  1. Install dependencies.

    To use the Clang compiler on macOS, you need to install an extra library for supporting OpenMP. You can use brew to install this and other dependencies.

    brew install libomp

  2. Then install a BLAS implementation; OpenBLAS is the default choice.

    brew install openblas

    Next, install Xcode Command Line Tools.

    xcode-select --install

On Windows you need to use Anaconda3 or Miniconda3 to install all the dependencies.

  1. Install compiler requirements.

    conda install --update-deps vs2017_win-64 vs2017_win-32 msvc_runtime

  2. Install binary and build dependencies.

    conda install --update-deps -c conda-forge -y openblas cmake

  1. Build and install qiskit-aer directly

    If you have pip <19.0.0 installed and your environment doesn’t require a custom build, run:

    cd qiskit-aer
pip install .

    This will both build the binaries and install Aer.

    Alternatively, if you have a newer pip installed, or have some custom requirement, you can build a Python wheel manually.

    cd qiskit-aer
python ./setup.py bdist_wheel

    If you need to set a custom option during the wheel build, refer to Custom options during wheel builds.

    After you build the Python wheel, it will be stored in the dist/ dir in the Aer repository. The exact version will depend

    cd dist
pip install qiskit_aer-*.whl

    The exact filename of the output wheel file depends on the current version of Aer under development.

Custom options during wheel builds¶

The Aer build system uses scikit-build to run the compilation when building it with the Python interface. It acts as an interface for setuptools to call CMake and compile the binaries for your local system.

Due to the complexity of compiling the binaries, you may need to pass options to a certain part of the build process. The way to pass variables is:

python setup.py bdist_wheel [skbuild_opts] [-- [cmake_opts] [-- build_tool_opts]]

where the elements within square brackets [] are optional, and skbuild_opts, cmake_opts, build_tool_opts are to be replaced by flags of your choice. A list of CMake options is available here: https://cmake.org/cmake/help/v3.6/manual/cmake.1.html#options. For example, you could run something like:

python setup.py bdist_wheel -- -- -j8

This is passing the flag -j8 to the underlying build system (which in this case is Automake), telling it that you want to build in parallel using 8 processes.

For example, a common use case for these flags on linux is to specify a specific version of the C++ compiler to use (normally if the default is too old):

python setup.py bdist_wheel -- -DCMAKE_CXX_COMPILER=g++-7

which will tell CMake to use the g++-7 command instead of the default g++ when compiling Aer.

Another common use case for this, depending on your environment, is that you may need to specify your platform name and turn off static linking.

python setup.py bdist_wheel --plat-name macosx-10.9-x86_64 \
-- -DSTATIC_LINKING=False -- -j8

Here --plat-name is a flag to setuptools, to specify the platform name to use in the package metadata, -DSTATIC_LINKING is a flag for using CMake to disable static linking, and -j8 is a flag for using Automake to use 8 processes for compilation.

A list of common options depending on platform are:

Platform

Tool

Option

Use Case

All

Automake

-j

Followed by a number, sets the number of processes to use for compilation.

Linux

CMake

-DCMAKE_CXX_COMPILER

Used to specify a specific C++ compiler; this is often needed if your default g++ is too old.

OSX

setuptools

–plat-name

Used to specify the platform name in the output Python package.

OSX

CMake

-DSTATIC_LINKING

Used to specify whether or not static linking should be used.

Nota

Some of these options are not platform-specific. These particular platforms are listed because they are commonly used in the environment. Refer to the tool documentation for more information.
