Kontribusi untuk Qiskit¶

Qiskit adalah sebuah proyek sumber terbuka yang berkomitmen untuk membawakan komputasi kuantum kepada orang-orang dengan latar belakang apapun. Halaman ini menjelaskan bagaimana cara agar Anda dapat bergabung dengan komunitas Qiskit dalam menyukseskan tujuan ini.

Tempat Menemukan Kode¶

Kode sumber untuk Qiskit ada di GitHub organisasi Qiskit, dimana Anda dapat menemukan proyek-proyek individu yang membentuk Qiskit, termasuk

  • Qiskit Terra

  • Qiskit Aer

  • Qiskit Ignis

  • Qiskit Aqua

  • Layanan IBMQ Qiskit (IBMQ Provider)

  • Tutorial Qiskit

  • Dokumentasi Antarmuka Pemrograman Aplikasi (API) Qiskit

Memulai¶

Cari tahu bagaimana para anggota komunitas Qiskit

  • Hubungan antar satu sama lain

  • Mendiskusikan ide

  • Mencari bantuan ketika menemukan masalah

  • Mengikuti berita terbaru di komunitas

  • Menjaga konsistensi langgam kode

  • Membuat paket kode Qiskit dari sumber

Kontribusi Kode¶

Panduan Langgam¶

Untuk memastikan langgam kode yang konsisten dalam proyek, kami menggunakan Pylint dan pycodesytle untuk memverifikasi bahwa kontribusi kode sudah sesuai dan menghormati panduan langgam proyek. Untuk memverifikasi bahwa perubahan Anda sudah sesuai dengan panduan langgam, mohon jalankan: tox -elint

Perjanjian Lisensi Kontributor¶

Sebelum Anda dapat menyerahkan kode apa pun, semua kontributor harus menandatangani perjanjian lisensi kontributor (CLA). Dengan menandatangani CLA, Anda menegaskan bahwa Anda adalah penulis dari kontribusi, dan bahwa Anda mengkontribusikannya secara bebas di bawah persyaratan lisensi Apache-2.0.

Ketika Anda berkontribusi pada proyek Qiskit dengan permintaan tarik (pull request) baru, bot akan mengevaluasi apakah Anda telah menandatangani CLA tersebut. Jika diperlukan, bot akan memberi komentar pada permintaan tarik tersebut, beserta sebuah link untuk menerima perjanjian tersebut. Dokumen CLA individu tersedia jika Anda ingin meninjaunya (dalam format PDF).

Catatan

Jika kontribusi Anda adalah bagian dari pekerjaan Anda atau kontribusi Anda adalah properti dari atasan Anda, maka Anda kemungkinan besar perlu untuk menandatangani CLA perusahaan juga dan mengirimkannya kepada kami melalui surel di <qiskit@us.ibm.com>.

Permintaan Tarik (Pull Requests)¶

Kami menggunakan permintaan tarik (pull request) GitHub untuk menerima kontribusi.

Walaupun tidak diwajibkan, membuka isu baru tentang bug yang sedang Anda perbaiki atau fitur yang sedang Anda kerjakan sebelum Anda membuka permintaan tarik (pull request) adalah langkah penting dalam memulai diskusi dengan komunitas tentang pekerjaan Anda. Isu ini memberi kita tempat untuk berbicara tentang ide dan bagaimana kita dapat bekerja sama untuk menerapkannya dalam kode. Hal ini juga memungkinkan komunitas mengetahui apa yang sedang Anda kerjakan, dan jika Anda membutuhkan bantuan, Anda dapat mengacu pada isu tersebut ketika mendiskusikannya dengan komunitas dan anggota tim lainnya.

Jika Anda telah menulis sebagian kode tetapi perlu bantuan menyelesaikannya, ingin mendapatkan umpan balik dulu sebelum menyelesaikannya, atau ingin berbagi dan mendiskusikan terlebih dahulu sebelum menyelesaikan implementasinya, Anda dapat membuat sebuah permintaan tarik (pull request) Work in Progress. Ketika Anda membuat permintaan tarik, tambahkan judul dengan tag [WIP] (singkatan Work In Progress atau Pekerjaan Dalam Proses). Ini akan menunjukkan kepada peninjau kode bahwa kode dalam permintaan tarik belum dalam keadaan akhir dan masih akan berubah. Ini juga berarti bahwa kami tidak akan menggabungkan komit tersebut sampai kode tersebut selesai. Anda atau seorang peninjau dapat menghapus tag [WIP] ketika kode siap untuk ditinjau penuh untuk penggabungan.

Pesan Komit¶

Isi dari pesan komit yang mendeskripsikan perubahan sama pentingnya dengan perubahan itu sendiri. Pesan komit memberikan konteks tidak hanya untuk peninjauan kode tetapi juga riwayat perubahan dalam log git. Sebuah pesan yang rinci akan membuat kode Anda lebih mudah untuk ditinjau, dan juga akan memberikan konteks atas perubahan tersebut ketika seseorang melihatnya di masa depan. Ketika menulis pesan komit, ingat detail penting ini:

Jangan menganggap peninjau mengerti apa masalah aslinya.

Ketika membaca suatu isu, setelah sejumlah komentar bolak-balik, sering kali jelas apa yang menjadi akar permasalahan. Pesan komit harus memiliki pernyataan yang jelas mengenai apa yang menjadi masalah aslinya. Bug-nya hanyalah sekedar latar belakang sejarah yang menarik pada * bagaimana * masalah teridentifikasi. Haruslah memungkinkan untuk meninjau tambalan kode yang diusulkan hanya dari pesan komit, tanpa perlu membaca tiket bug-nya.

Jangan menganggap bahwa kode itu sendiri sudah cukup jelas tanpa perlu adanya dokumentasi.

Apa yang terlihat jelas bagi satu orang, mungkin tidak jelas bagi orang lain. Selalu dokumentasikan apa masalah aslinya dan bagaimana masalah itu diperbaiki, untuk setiap perubahan kecuali kesalahan ketik yang paling jelas terlihat, atau komit yang hanya whitespace.

Deskripsikan mengapa sebuah perubahan dibuat.

Hanya mendokumentasikan bagaimana kode ditulis tanpa menjelaskan mengapa pengembang memilih untuk melakukannya dengan cara tersebut adalah sebuah kesalahan umum. Tentu saja, Anda harus mendeskripsikan struktur kode secara keseluruhan, terutama untuk perubahan besar, tetapi yang lebih penting, pastikan untuk mendeskripsikan maksud/motivasi di balik perubahan tersebut.

Baca pesan komit untuk melihat apakah pesan tersebut menunjukkan peningkatan struktur kode.

Seringkali ketika mendeskripsikan pesan komit yang besar, terlihat jelas bahwa komit seharusnya dipecah menjadi dua bagian atau lebih. Jangan takut untuk kembali dan menerapkan kembali (rebase) perubahan tersebut untuk membaginya menjadi pull request terpisah.

Pastikan bahwa informasi memadai untuk ditinjau.

Ketika GitHub mengirimkan notifikasi email untuk pengajuan permintaan tarik (pull request) baru, ada sebagian informasi yang disertakan - biasanya hanya pesan komit (commit) dan daftar file yang termodifikasi. Karena volume perubahan (patch) yang tinggi, sebuah pesan komit harus mengandung informasi yang cukup untuk para pengulas untuk menemukan perubahan yang perlu mereka tinjau.

Baris komit pertama adalah yang paling penting.

Dalam komit Git, baris pertama dari pesan komit memiliki signifikansi khusus. Baris ini digunakan sebagai judul pull request lazim, baris subjek pada notifikasi surel, anotasi pesan git, anotasi penampil gitk, menggabungkan pesan komit, dan di banyak tempat lainnya yang ruangnya terbatas. Baris ini juga meringkas perubahan itu sendiri, baris ini harus mampu menjelaskan detail dari bagian kode mana yang terpengaruh.

Selain itu, baris pertama dari pesan komit menjadi entri dalam changelog yang dihasilkan jika PR ditandai untuk dimasukkan ke dalam changelog. Sangat penting bagi Anda untuk menulis baris ini dengan jelas dan ringkas.

Deskripsikan batasan dari kode yang ada saat ini.

Jika kode yang sedang diubah masih bisa dikembangkan, atau masih ada batasan tertentu yang diketahui, sebutkan ini dalam pesan commit. Hal ini menunjukkan kepada peninjau bahwa gambaran umum telah dipertimbangkan, dan apa tradeoff yang telah dilakukan dengan mempertimbangkan tujuan jangka pendek dan keinginan jangka panjang.

Sertakan referensi untuk masalahnya.

Jika perbaikan commit berhubungan dengan suatu masalah, pastikan Anda menganotasinya dalam pesan commit. Gunakan sintaks:

Fixes #1234

jika ini memperbaiki masalah (GitHub akan menutup masalah ketika PR digabungkan).

Aturan utama yang harus diikuti adalah:

Pesan commit harus berisi semua informasi yang diperlukan untuk sepenuhnya memahami dan meninjau kebenaran dari patch. Kurang bukan berarti lebih.

Mendokumentasikan Kode Anda¶

Jika Anda membuat perubahan pada sebuah elemen, pastikan Anda memperbarui docstrings yang terkait dan bagian-bagian dari dokumentasi pada docs/apidocs di repo yang sesuai. Untuk membangun secara lokal dokumentasi yang spesifik untuk setiap elemen, jalankan tox -edocs untuk mengkompilasi dan membangun dokumentasi secara lokal dan simpan keluaran ke docs/_build/html. Selain itu, pekerjaan CI Docs pada azure pipelines akan menjalankan ini dan meng-host sebuah file zip dari keluaran yang dapat Anda unduh dan lihat secara lokal.

Jika Anda memiliki masalah dengan kombinasi dokumentasi yang dikelola di Qiskit/qiskit repo, Anda dapat membuka dokumentasi edisi jika Anda melihat bug pada doc, atau menemukan fitur baru yang perlu didokumentasikan, atau berpikir bahwa materi bisa ditambahkan ke docs yang ada.

Kontribusi pertama yang baik¶

Jika Anda ingin berkontribusi pada Qiskit, namun tidak yakin di mana harus memulai, label good first issue pada isu untuk sebuah proyek menandakan isu-isu yang sesuai untuk kontributor baru. Ini adalah isu-isu yang telah ditinjau dan ditandai oleh para kontributor sebagai isu yang dapat diselesaikan kontributor baru. Dengan kata lain, pengetahuan mendalam mengenai Qiskit tidaklah diperlukan untuk menyelesaikan masalah tersebut.

Kebijakan Deprekasi¶

Pengguna Qiskit perlu mengetahui apakah sebuah fitur atau API yang mereka andalkan akan terus didukung oleh perangkat lunak di masa depan. Penting bagi pengguna untuk dapat mengetahui di bawah kondisi apa proyek dapat menghapus (atau mengubah secara permanen) fitur atau API. Untuk membatasi ekspektasi, kebijakan berikut ini adalah bagaimana API dan fitur depekkrasi/penghapusan ditangani oleh Qiskit:

1. Features, APIs, or configuration options are marked deprecated in the code. Appropriate DeprecationWarning class warnings will be sent to the user. The deprecated code will be frozen and receive only minimal maintenance (just so that it continues to work as-is).

2. A migration path will be documented for current users of the feature. This will be outlined in the both the release notes adding the deprecation, and the release notes removing the feature at the completion of the deprecation cycle. If feasible, the warning message will also include the migration path. A migration path might be "stop using that feature", but in such cases it is necessary to first judge how widely used and/or important the feature is to users, in order to determine a reasonable obsolescence date.

3. An obsolescence date for the feature will be set. The feature must remain intact and working (although with the proper warning being emitted) in all releases pushed until after that obsolescence date. At the very minimum, the feature (or API, or configuration option) should be marked as deprecated (and continue to be supported) for at least three months of linear time from the release date of the first release to include the deprecation warning. For example, if a feature were deprecated in the 0.9.0 release of Terra, which was released on August 22, 2019, then that feature should still appear in all releases until at least November 22, 2019. Since releases do not occur at fixed time intervals, a deprecation warning may only occur in one release prior to removal.

Perhatikan bahwa keterlambatan ini adalah minimum. Untuk fitur yang signifikan, disarankan bahwa fitur terdeprekasi muncul untuk setidaknya dua kali lipat waktu itu. Selain itu, per kebijakan cabang yang stabil, penghapusan deprekasi hanya dapat untuk rilis versi minor; mereka tidak cocok untuk backporting.

Peringatan Deprecation¶

Cara yang tepat untuk menaikkan peringatan deprecation adalah dengan menggunakan fungsi warn dari modul warning module di pustaka standar Python. Kelas kategori peringatan harus berupa DeprecationWarning. Contohnya:

import warnings

def foo(input):
    warnings.warn('The qiskit.foo() function is deprecated as of 0.9.0, and '
                  'will be removed no earlier than 3 months after that '
                  'release date. You should use the qiskit.bar() function '
                  'instead.', DeprecationWarning, stacklevel=2)

Satu hal yang perlu diperhatikan di sini adalah kwarg stack_level pada panggilan warn(). Argumen ini digunakan untuk menentukan level mana pada stack call yangakan digunakan sebagai baris yang menginisiasi peringatan. Biasanya, stack_level harus ditetapkan pada angka 2, karena ini akan menunjukkan garis yang memanggil konteks di mana peringatan diberikan. Dalam contoh di atas, terjadi pada pemanggil foo(). Jika Anda tidak mengatur hal ini, peringatan akan menunjukkan bahwa hal itu disebabkan oleh baris dalam fungsi foo(), yang tidak berguna bagi pengguna yang mencoba untuk menentukan asal dari panggilan yang terdeprekasi. Namun, nilai ini dapat disesuaikan, tergantung pada panggilan stack dan dari mana warn() akan dipanggil. Sebagai contoh, jika peringatan selalu dinaikkan dengan metode pribadi yang hanya memiliki satu pemanggil, stack_level=3 mungkin tepat.

Kebijakan Cabang Stabil¶

Cabang yang stabil dimaksudkan untuk menjadi sumber aman untuk perbaikan bagi bug dampak tinggi dan masalah keamanan yang telah diperbaiki pada master sejak rilisnya. Ketika meninjau PR cabang stabil, kita harus menyeimbangkan risiko dari setiap patch yang diberikan dengan nilai yang akan diberikan nya kepada pengguna dari cabang stabil. Hanya beberapa kelas tertentu dari perubahan yang sesuai untuk dimasukkan pada cabang stabil. Sebuah patch yang besar dan berisiko untuk isu yang besar mungkin masuk akal, juga perbaikan sepele untuk kasus penanganan error yang kurang diketahui. Sejumlah faktor harus dipertimbangkan ketika ingin mengajukan perubahan:

  • Risiko regresi: perubahan terkecil sekalipun berisiko merusak sesuatu, dan kita benar-benar ingin menghindari regresi pada cabang stabil.

  • Manfaat visibilitas pengguna: apakah kita memperbaiki sesuatu yang benar-benar pengguna sadari, dan apabila iya, seberapa penting hal itu?

  • Seberapa self-contained perbaikan tersebut: jika ia memperbaiki isu signifikan tetapi juga melakukan refaktor untuk banyak kode, mungkin perlu dipikirkan kembali apa perbaikan yang lebih tidak berisiko.

  • Apakah perbaikan sudah ada pada master: perubahan harus menjadi backport dari perubahan yang sudah digabungkan ke master, kecuali perubahan pada master tidak masuk akal.

Prosedur backporting:¶

Ketika meng-backport patch dari master ke stabil, kita ingin menyimpan referensi dari perubahan pada master. Ketika Anda membuat branch untuk PR stabil, gunakan:

$ git cherry-pick -x $master_commit_id

Namun, ini hanya bekerja untuk patch mandiri yang kecil dari master. Jika Anda perlu melakukan backport sebuah subset dari komit yang lebih besar (dari sebuah PR yang tertindih, misalnya) dari master, lakukan ini secara manual. Dalam kasus-kasus seperti ini, tambahkan:

Backported from: #master pr number

sehingga kita dapat melacak sumber dari subset perubahan, bahkan jika pemilihan yang ketat tidak masuk akal.

Jika patch yang anda ajukan tidak memilih yang sesuai secara bersih/keseluruhan, anda dapat membantu dengan menyelesaikan masalah anda sendiri dan mengajukan patch yang dihasilkan. Harap menjaga baris konflik pada pesan komit untuk membantu meninjau dari patch yang stabil.

Label backport¶

Bug-bug atau PR- PR yang ditandai dengan stable backport potential adalah bug-bug yang digunakan juga untuk pelepasan yang stabil dan mungkin cocok untuk pengimporan kembali setelah perbaikan berada di master. Setelah backport telah diusulkan, tanda yang ada harus dihapus.

Gunakan [Stable] di dalam judul PR terhadap cabang yang stabil, sebagai penanda bahwa menetapkan cabang target sebagai stabil bukan merupakan suatu kesalahan. Cantumkan juga referensi nomor PR di bagian master yang Anda sedang porting.

Berkontribusi Pada Dokumentasi¶

Dokumentasi Qiskit dibentuk oleh filosofi docs sebagai kode, yang utamanya ditarik dari kode komentar Qiskit di dalam gaya dokumentasi API.

Dokumentasi ini dibangun dari cabang master dari Qiskit/qiskit/docs menggunakan Sphinx. Sebagian besar dokumentasi, berdasarkan API Reference, diambil dari kode-kode komentar dalam repositori yang tercantum dalam Tempat Menemukan Kode.

Struktur Dokumentasi¶

Dokumentasi di dalam Qiskit terstruktur dengan cara yang akan mendorong sebanyak mungkin dokumentasi yang sebenarnya ke dalam docstrings. Hal ini mempermudah pengguna untuk melakukan penambahan dan koreksi saat pengembangan, karena mayoritas dokumentasi bersifat dekat dengan kode yang sedang diubah. Ada tiga tingkatan dalam struktur dokumentasi normal di Terra:

File .rst di dalam docs/apidocs

File-file ini digunakan untuk memberitahu Sphinx modul manakah yang perlu diikutsertakan dalam dokumentasi yang dirender. Di dalamnya terdapat dua informasi: internal reference atau cross reference kepada modul, yang dapat digunakan untuk sambungan internal di dalam dokumentasi, dan automodule directive yang digunakan untuk menguraikan modul docstrings dari path import tertentu. Contohnya, file dagcircuit.rst mengandung:

.. _qiskit-dagcircuit:


.. automodule:: qiskit.dagcircuit
   :no-members:
   :no-inherited-members:
   :no-special-members:

Satu-satunya file .rst yang ada di luarnya adalah qiskit.rst, yang berisi daftar isi. Jika Anda menambahkan file .rst baru untuk dokumentasi modul baru, pastikan untuk menambahkannya ke toctree di dalam file tersebut.

Docstring pada tingkatan modul

Docstring ini berada pada tingkat modul untuk modul yang dinyatakan dalam direktif automodule dalam file rst. Jika modul tersebut adalah sebuah direktori/ruangnama, docstring tersebut perlu dirincikan dalah file __init__.py untuk direktori tersebut. Docstring tingkat modul ini mengandung lebih banyak detil mengenai modul yang sedang didokumentasikan. Struktur normal docstring ini adalah untuk menjabarkan seluruh kelas dan fungsi API publik yang terkandung dalam modul tersebut. Hal ini biasa dilakukan dengan menggunakan autosummary directive (atau autodoc directives secara langsung jika modul tersebut sederhana, seperti pada kasus qiskit.execute). Direktif autosummary tersebut digunakan untuk mendokumentasikan otomatis sebuah daftar berisi elemen Python yang berbeda (kelas, fungsi, dll.) secara langsung tanpa perlu memanggil direktif autodoc masing-masing secara manual. Docstring tingkat modul adalah tempat untuk memberikan gambaran tingkat tinggi mengenai fungsionalitas yang disediakan modul tersebut. Hal ini biasa dilakukan dengan mengelompokkan komponen-komponen yang berbeda dari API publik ke dalam beberapa subbagian.

Sebagai contoh, seperti pada contoh modul dagsirkuit sebelumnya, isi dari modul docstring untuk qiskit/dagcircuit/__init__.py akan:

"""
=======================================
DAG Circuits (:mod:`qiskit.dagcircuit`)
=======================================
.. currentmodule:: qiskit.dagcircuit
DAG Circuits
============
.. autosummary::
   :toctree: ../stubs/
   DAGCircuit
   DAGNode
Exceptions
==========
.. autosummary::
   :toctree: ../stubs/
   DAGCircuitError
"""

Catatan

Ini hanyalah sebuah contoh dan modul docstring yang sebenarnya untuk modul dagsirkuit mungkin menyimpang dari ini.

Docstring yang sebenarnya untuk elemen tercantum dalam modul docstring

Anda harus berusaha untuk mendokumentasikan secara menyeluruh semua tampilan antarmuka publik yang terekspos menggunakan contoh ketika diperlukan. Untuk docstrings, Google Python Style Docstrings sudah digunakan. Hal ini diurai menggunakan ekstensi napoleon sphinx. dokumentasi napoleon memuat contoh yang benar tentang bagaimana docstrings harus diformat.

Catatan

Anda dapat menggunakan direktif Sphinx atau format rst dalam sebuah docstring apa saja selama masuk akal. Sebagai contoh, sebuah ekstensi yang umum digunakan adalah direktif jupyter-execute, yang digunakan untuk menjalankan sebuah blok kode di Jupyter dan menampilkan kode dan keluaran. Hal ini sangat berguna untuk visualisasi.

Integrasi dokumentasi¶

Dokumentasi yang tersimpan pada https://qiskit.org/documentation/ mencakup seluruh proyek Qiskit; Terra merupakan salah satu komponen tersebut. Dengan demikian, rangka (build) dokumentasi untuk versi yang disimpan dibangun oleh repositori meta-paket Qiskit https://github.com/Qiskit/qiskit. Ketika pengerjaan (commit) digabungkan ke repositori tersebut, keluaran rangka Sphinx diunggah ke website qiskit.org. Rangka-rangka Sphinx tersebut dikonfigurasikan untuk menarik dokumentasi dari versi elemen Qiskit yang diinstal oleh meta-paket pada saat itu. Sebagai contoh, jika versi meta-paket saat ini adalah 0.13.0, maka itu akan menyalin dokumentasi dari keluaran Terra 0.10.0. Ketika persyaratan meta-paket dinaikkan, maka itu akan mulai menarik dokumentasi dari versi yang baru. Hal ini berarti perbaikan untuk dokumentasi API yang salah akan perlu diikutsertakan dalam rilis baru. Perbaikan dokumentasi adalah backport yang valid untuk rilis patch yang stabil mengikuti kebijakan cabang stabil (lihat bagian tersebut di bawah).

Selama proses pembuatan rangka (build), isi masing-masing elemen docs/apidocs/ secara rekursif disalin ke dalam sebuah salinan bersama dari doc/apidocs/ dalam repositori meta-paket bersama dengan semua elemen lainnya. Hal ini berarti apa yang berada dalam akar docs/apidocs masing-masing elemen pada sebuah keluaran akan berakhir pada akar dari https://qiskit.org/documentation/apidoc/.

Menerjemahkan Dokumentasi¶

Dokumentasi Qiskit diterjemahkan (terlokalisasi) menggunakan Crowdin, sebuah perangkat lunak dan platform lokalisasi web yang memungkinkan organisasi untuk mengoordinasikan proyek penerjemahan dan berkolaborasi dengan komunitas untuk menerjemahkan materi. Crowdin memungkinkan komunitas para penerjemah kita untuk memperkuat dampak mereka dengan secara otomatis menggunakan kembali karya yang diinvestasikan untuk menerjemahkan satu kalimat untuk menerjemahkan kalimat lain yang serupa. Crowdin juga membuat ketahanan terjemahan terhadap banyak jenis perubahan pada material aslinya, seperti memindahkan kalimat ke sekitarnya, bahkan dapat melintasi antar berkas.

Permintaan lokalisasi Qiskit ditangani dalam repositori Qiskit Translations. Untuk berkontribusi pada lokalisasi Qiskit, harap ikuti langkah-langkah berikut ini:

  1. Tambahkan nama Anda (atau ID) ke LOCALIZATION_KONTRIBUTOR file.

  2. Buat pull request (PR) untuk menggabungkan perubahan Anda. Pastikan untuk mengikuti contoh untuk membuka Pull request.

    Catatan

    • Setiap kontributor harus membuat PR mereka sendiri dan menyetujui CLA.

    • Harap menyebutkan bahasa yang anda inginkan untuk berkontribusi di dalam ringkasan PR.

    • Jika Anda memiliki masalah terkait permintaan bahasa, tambahkan tautan masalah ke PR.

  3. Anda akan diminta untuk menandatangani Perjanjian Lisensi Kontributor Qiskit (Qiskit Contributor License Agreement-"CLA"); harap lakukan hal tersebut.

  4. Minimal tiga kontributor per bahasa diperlukan untuk setiap bahasa baru yang akan ditambahkan, untuk menerima dukungan resmi dari pengurus proyek lokalisasi.

  5. Di antara kelompok kontributor, pemimpin penerjemah harus diidentifikasi yang berfungsi sebagai penghubung dengan para pengurus proyek setempat. Pemimpin ini harus menghubungi: Yuri Kobayashi (yurik@jp.ibm.com) melalui email.

  6. Dalam proyek Qiskit-Docs Crowdin, pilih bahasa yang ingin Anda kontribusikan.

    Catatan

    Seperti yang disebutkan dalam pos blog Qiskit dalam bahasa saya adalah Qiskit, kami ingin memastikan bahwa bahasa terjemahan memiliki dukungan komunitas yang cukup untuk membangun sebuah tim penerjemah dengan penerjemah, korektor, dan pimpinan terjemahan. Jika Anda ingin menjadi seorang pimpinan terjemahan atau bersedia untuk bergabung dengan sebuah tim proyek penerjemahan baru, Anda dapat membuka sebuah isu GitHub untuk memulai sebuah diskusi dengan tim Qiskit dan merekrut anggota proyek penerjemahan.

  7. Klik tombol Join dan paste URL dari PR Anda di kotak dialog di mana Anda ditanya mengapa Anda ingin bergabung dengan proyek Crowdin.

Para administrator proyek Crowdin akan meninjau permintaan Anda dan memberi Anda akses secepat yang mereka bisa.

Membangun dari Sumber¶

Anda dapat membuat salinan lokal dokumentasi ini dari kloningan lokal milik Anda untuk repositori Qiskit/qiskit sebagai berikut:

  1. Kloning repositori Qiskit.

    git clone https://github.com/Qiskit/qiskit.git

  2. Mengkloning repositori akan membuat sebuah folder lokal bernama qiskit.

    cd qiskit

  3. Bangun dokumentasi dengan menuju ke klon lokal Qiskit/qiskit Anda dan jalankan perintah berikut dalam sebuah jendela terminal.

    tox -edocs

    Jika Anda belum menginstal perintah tox, instal dengan menjalankan:

    pip install tox

Saat Anda membuat perubahan pada file RST lokal Anda, Anda dapat memperbarui file HTML Anda dengan menuju ke /doc/ dan menjalankan dalam jendela terminal seperti berikut:

tox -edocs

Ini akan membangun sebuah versi HTML dari repositori dokumentasi lokal Anda dalam subdirektori /docs/_build/html/.

Memasang dari sumber¶

Menginstal elemen dari sumber memungkinkan Anda untuk mengakses versi Qiskit terbaru alih-alih menggunakan versi yang terdapat dalam repositori Python Package Index (PyPI). Ini akan memampukan Anda untuk memeriksa dan memperpanjang versi terkini dari kode Qiskit secara lebih efisien.

Saat menginstal elemen dan komponen dari sumber, secara default versi pengembangan (yang sesuai dengan cabang git master) akan digunakan, sebagai ganti versi stabil (yang mengandung dasarkode (codebase) yang sama dengan paket pip yang terpublikasi). Karena versi pengembangan dari sebuah elemen atau komponen biasanya mengandung fitur dan perubahan baru, mereka umumnya juga memerlukan penggunaan versi pengembangan pada hal-hal sisanya.

Catatan

Paket Terra dan Aer membutuhkan kompiler untuk membangun dari sumber sebelum Anda dapat melakukan instalasi. Ignis, Aqua, dan backend Penyedia Kuantum IBM tidak memerlukan kompiler.

Memasang elemen dari sumber membutuhkan urutan instalasi yang diperintahkan untuk mencegah pemasangan dari versi elemen yang mungkin lebih rendah dari yang diinginkan jika versi pip merupakan versi sebelum versi sumber:

  1. qiskit-terra

  2. qiskit-aer

  3. qiskit-ignis

  4. qiskit-aqua

  5. qiskit-ibmq-provider (jika Anda ingin terhubung dengan perangkat Kuantum atau simulator online IBM)

Untuk bekerja dengan beberapa komponen dan elemen secara bersamaan, gunakan langkah-langkah berikut untuk setiap elemen.

Catatan

Penggunaan packaging namespace di Python menyebabkan perlunya perhatian terhadap bagaimana cara Anda menginstall packages. Jika Anda berencana untuk menginstall elemen manapun melalui sumber, jangan gunakan meta-package qiskit. Selain itu, ikuti petunjuk ini dan gunakan lingkungan virtual yang terpisah untuk melakukan pengembangan. Jika Anda memilih untuk menggabungkan instalasi yang telah ada dengan pengembangan Anda, silahkan lihat https://github.com/pypa/sample-namespace-packages/blob/master/table.md untuk serangkaian langkah instalasi yang tepat.

Mengatur Lingkungan Pengembangan Virtual (Virtual Development Environment)¶

conda create -y -n QiskitDevenv python=3
conda activate QiskitDevenv

Menginstal Terra dari Sumber¶

Menginstal dari sumber mensyaratkan bahwa Anda memiliki C++ compiler pada sistem Anda yang mendukung C + + 11.

Pada sebagian besar platform Linux, GCC compiler yang diperlukan sudah terinstal.

Jika Anda menggunakan macOS, Anda dapat menginstal Compiler Clang dengan menginstal XCode. Periksa apakah Anda memiliki XCode dan Clang yang terinstal dengan membuka jendela terminal dan masukkan perintah berikut.

clang --version

Instal XCode dan Clang dengan menggunakan perintah berikut.

xcode-select --install

Pada Windows, paling mudah untuk menginstal Visual C++ compiler dari Build Tools for Visual Studio 2017. Anda dapat menginstal Visual Studio versi 2015 atau 2017, pastikan untuk memilih pilihan untuk menginstal C++ compiler.

Setelah Compiler diinstal, Anda siap untuk menginstal Qiskit Terra.

  1. Mengkloning repositori Terra.

    git clone https://github.com/Qiskit/qiskit-terra.git

  2. Mengkloning repositori akan membuat sebuah folder lokal bernama qiskit-terra.

    cd qiskit-terra

  3. Instal pustaka Python yang dibutuhkan dari direktori qiskit-terra Anda.

    pip install cython

  4. Jika Anda ingin menjalankan tes atau melakukan pengecekan dengan linter, instal persyaratan pengembang.

    pip install -r requirements-dev.txt

  5. Instal qiskit-terra.

    pip install .

Jika Anda ingin memasangnya pada mode pengeditan, itu berarti perubahan kode pada proyek tidak memerlukan instalasi ulang:

pip install -e .

Anda kemudian dapat menjalankan contoh kode setelah menginstal Terra. Anda dapat menjalankan contoh dengan perintah berikut.

python examples/python/using_qiskit_terra_level_0.py

Catatan

Jika Anda tidak menginginkan untuk menginstall komponen lain, qiskit-terra akan mengeluarkan peringatan RuntimeWarning yang menyatakan bahwa qiskit-aer dan qiskit-ibmq-provider tidak terinstall. Peringatan ini muncul karena biasanya pengguna berniat untuk menggunakan elemen-elemen tambahan, namun tidak menyadari bahwa elemen-elemen tersebut belum terinstall, atau instalasi Aer atau IBM Quantum Provider ternyata gagal. Jika Anda ingin menghilangkan peringatan ini, tambahkan:

import warnings
warnings.filterwarnings('ignore', category=RuntimeWarning,
                        module='qiskit')

sebelum meng-import qiskit di dalam kode Anda. Hal tersebut akan menghilangkan peringatan tentang qiskit-aer dan qiskit-ibmq-provider yang tidak tersedia, namun akan tetap menampilkan peringatan-peringatan lain dari qiskit atau paket lainnya.

Menginstal Aer dari Sumber¶

  1. Mengkloning repositori Aer.

    git clone https://github.com/Qiskit/qiskit-aer

  2. Instal compiler yang dibutuhkan.

    pip install cmake scikit-build cython

Setelah ini, langkah-langkah untuk menginstal Aer bergantung pada sistem operasi yang Anda gunakan. Karena Aer adalah sebuah program C++ kompilasi dengan antarmuka Python, ada ketergantungan non-Python untuk membangun biner Aer yang tidak dapat diinstal secara universal bergantung pada sistem operasi.

  1. Install compiler requirements.

    Building Aer requires a C++ compiler and development headers.

    If you're using Fedora or an equivalent Linux distribution, install using:

    dnf install @development-tools

    For Ubuntu/Debian install it using:

    apt-get install build-essential

  2. Install OpenBLAS development headers.

    If you're using Fedora or an equivalent Linux distribution, install using:

    dnf install openblas-devel

    For Ubuntu/Debian install it using:

    apt-get install libopenblas-dev

  1. Install dependencies.

    To use the Clang compiler on macOS, you need to install an extra library for supporting OpenMP. You can use brew to install this and other dependencies.

    brew install libomp

  2. Then install a BLAS implementation; OpenBLAS is the default choice.

    brew install openblas

    Next, install Xcode Command Line Tools.

    xcode-select --install

On Windows you need to use Anaconda3 or Miniconda3 to install all the dependencies.

  1. Install compiler requirements.

    conda install --update-deps vs2017_win-64 vs2017_win-32 msvc_runtime

  2. Install binary and build dependencies.

    conda install --update-deps -c conda-forge -y openblas cmake

  1. Build dan install qiskit-aer secara langsung

    Jika Anda telah menginstal pip <19.0.0 dan environment Anda tidak membutuhkan build khusus (custom), jalankan:

    cd qiskit-aer
pip install .

    Hal ini akan mem-build binaries dan meng-install Aer.

    Sebagai alternatif, jika Anda memiliki pip versi yang lebih baru, atau memiliki persyaratan khusus (custom), Anda dapat mem-build Python wheel secara manual.

    cd qiskit-aer
python ./setup.py bdist_wheel

    Jika Anda perlu menetapkan opsi khusus selama wheel build, harap mengacu pada Opsi khusus selama pengaktifan wheel.

    Setelah Anda membangun Python wheel, maka akan disimpan dalam dist/ dir di repository Aer. Versi yang tepat akan tergantung

    cd dist
pip install qiskit_aer-*.whl

    Nama lengkap dari file output wheel tergantung dari versi Aer terbaru.

Opsi khusus selama pengaktifan wheel¶

Aer build system menggunakan scikit-build untuk menjalankan kompilasi ketika membangunnya dengan antarmuka Python. Ini bertindak sebagai antarmuka untuk setuptools untuk memanggil CMake dan compile biner untuk sistem lokal Anda.

Karena kompleksitas compiler biner, Anda mungkin perlu melewatkan pilihan pada bagian tertentu dari proses pembuatan. Cara untuk melewati variabel adalah:

python setup.py bdist_wheel [skbuild_opts] [-- [cmake_opts] [-- build_tool_opts]]

dimana elemen dalam kurung persegi [] adalah opsional, dan skbuild_opts, cmake_opts, build_tool_opts akan digantikan oleh bendera dari pilihan Anda. Daftar opsi CMake tersedia di sini: https://cmake.org/cmake/help/v3.6/manual/cmake.1.html#options. Sebagai contoh, Anda dapat menjalankan sesuatu seperti:

python setup.py bdist_wheel -- -- -j8

Hal ini melewati bendera -j8 ke sistem build yang mendasari (yang dalam hal ini adalah Automake), mengatakan bahwa Anda ingin membangun secara paralel dengan menggunakan 8 proses.

Sebagai contoh, kasus penggunaan umum untuk flag ini pada linux adalah untuk menentukan versi spesifik dari compiler C++ untuk digunakan (biasanya jika default nya terlalu tua):

python setup.py bdist_wheel -- -DCMAKE_CXX_COMPILER=g++-7

yang akan memberitahu CMake untuk menggunakan perintah g + + -7 bukan default g + + saat melakukan compile Aer.

Kasus penggunaan lain yang umum untuk ini, tergantung pada lingkungan Anda, adalah bahwa Anda mungkin perlu untuk menentukan nama platform Anda dan mematikan penghubung statis.

python setup.py bdist_wheel --plat-name macosx-10.9-x86_64 \
-- -DSTATIC_LINKING=False -- -j8

--plat-name di sini adalah penanda bagi setuptools, untuk memberikan nama platform untuk digunakan pada metadata package, -DSTATIC_LINKING adalah penanda untuk menggunakan CMake untuk menonaktifkan static linking, dan -j8 adalah penanda untuk menggunakan Automake untuk menggunakan 8 proses untuk kompilasi.

Daftar opsi umum yang tergantung pada platform adalah:

Platform

Alat

Opsi

Use Case

Semua

Automake

-j

Diikuti dengan angka, atur jumlah proses yang akan digunakan untuk melakukan kompilasi.

Linux

CMake

-DCMAKE_CXX_COMPILER

Digunakan untuk mendefinisikan kompiler C++ tertentu; hal ini sering dibutuhkan jika g++ bawaan yang Anda miliki terlalu tua.

OSX

setuptools

-- plat-name

Digunakan untuk menentukan nama platform dalam paket keluaran Python.

OSX

CMake

-DSTATIC_LINKING

Digunakan untuk menentukan apakah linking statis perlu dipakai.

Catatan

Beberapa pilihan ini tidak spesifik terhadap platform yang digunakan. Platform-platform tertentu ini terdaftar karena mereka biasa digunakan pada environment. Silahkan mengacu kepada dokumentasi tool untuk informasi lebih lanjut.
